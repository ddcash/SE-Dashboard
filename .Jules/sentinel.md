## 2024-10-09 - [Stored & DOM XSS via Unsafe Bookmark URLs]
**Vulnerability:** Bookmark URLs were output directly into `href` attributes in `<a href="${esc(bm.url)}">` and interpolated directly into inline event handlers in the command palette (`onclick="window.open('${esc(bm.url)}',...)"`), leading to Stored XSS via dangerous protocols (e.g. `javascript:alert(1)`) and DOM XSS via unescaped string literal interpolation.
**Learning:** `esc` only HTML-encodes attributes, but does not sanitize dangerous protocols (like `javascript:`, `vbscript:`, `data:`). In addition, even if a URL is HTML escaped, interpolating it directly into an inline JavaScript string literal in an `onclick` attribute can allow DOM XSS if the string contains unescaped quotes or if the execution context is triggered.
**Prevention:** Always validate and sanitize URLs against an allow-list or block-list of dangerous protocols before setting `href`. For dynamic JS inline handlers, avoid string interpolation altogether; instead, store data in a global array or data-attributes and reference it via index or DOM traversal.
## 2026-07-08 - Fix XSS via string interpolation in form onsubmit handlers
**Vulnerability:** User-controlled inputs (like group or category IDs) were interpolated directly into inline `onsubmit` handlers (e.g., `onsubmit="submitGroup(event, '${groupId}')"`). An attacker could manipulate these IDs to break out of the string context and execute arbitrary JS, bypassing standard HTML escaping since browsers decode HTML entities in attributes before executing JS.
**Learning:** Interpolating user-supplied strings directly into inline JS event handlers (like `onsubmit`) is unsafe regardless of HTML escaping.
**Prevention:** Store user-provided strings in `data-*` attributes (e.g., `data-group-id`) and retrieve them dynamically within the event handler (using `e.target.dataset`) rather than interpolating them directly into the inline script.
